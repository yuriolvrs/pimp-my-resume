# Phase 4 Plan — Resume generation

Status: implemented (see PROGRESS.md's Phase 4 entries, 2026-07-24). Open decisions below were resolved with the user before coding started. The "generation" flow item 2 originally described went through two revisions after being tried live: first replaced with one LLM call per resume section (after the whole-resume-in-one-call design proved unreliable), then replaced again entirely with a non-LLM selection/reordering approach (src/lib/generation/selectResumeContent.ts) after the per-section design was found, from an actual exported resume, to fabricate technical claims not present in the cited evidence. No LLM call is involved in resume generation at all anymore -- see PROGRESS.md for the full account.

## Context
PRD §6's original `Generation.sourceMap` (`{generatedText, profileEvidence: string[]}`, free text) and `JobAnalysis.matches` (`{requirement, profileEvidence: string[]}`) are already superseded by Phase 3's atom-based model: `ProfileAtom[]` (stable ids) + `RequirementMatch.atomIds` (references, never free text) — done specifically to make fabrication structurally impossible rather than just checked after the fact. Phase 4 should extend that same pattern rather than reintroduce free-text evidence: update `SourceMapEntry.profileEvidence: string[]` (free text) → `atomIds: string[]` (references into `ProfileAtom`), matching `RequirementMatch`'s shape. `ResumeContent` stays as-is (PRD-defined structured shape).

Reinforced by a concrete finding while hardening Phase 3's matching prompt: `matchRequirement.ts` originally also asked for a free-text `note` explaining its reasoning, constrained only by instructions ("ground this only in what's shown," "never invent a missing skill"). It fabricated anyway -- inventing a missing skill not named in the requirement, and separately asserting "3+ years of experience" confirmed when the only cited evidence was a bare `Skills: Java` tag with no duration information at all. The `note` field was removed entirely rather than re-patched, because nothing about free text can be checked; `atomIds`/`status` stayed because they're structurally constrained (ids must exist in the candidate list). Phase 4 should not reintroduce anything like `note` for the same reason.

## Proposed flow
1. **Entry point:** on `JobDetailPage`, once matching is confirmed (`analysis.matches` populated), a new "Generate resume" action appears alongside "View matches" — navigates to a new route `/jobs/:id/resume`.
2. **Generation call:** one LLM call, input = job analysis (role summary, requirements, keywords) + only the atoms already confirmed as evidence somewhere (`RequirementMatch.atomIds` union + `profile.additionalInfo`) + contact/summary/education (structural, not generated). Prompt instructs: select and rewrite bullets to emphasize fit, but every generated bullet must be traceable to one or more atom ids — same anti-fabrication contract as `matchRequirement.ts`, just applied to generation instead of verification.
3. **Output:** `ResumeContent` (contact/summary/skills/experience/projects/education) + `SourceMapEntry[]` (`generatedText` + `atomIds`). Parsed defensively via `generateStructured`, same as existing prompts.
4. **Storage:** new `src/lib/genStore.ts` mirroring `jobStore.ts`'s pattern (`newGeneration`, `loadGeneration`, `saveGeneration`, `listGenerationsForPosting`) against the already-declared `generations` Dexie table (unused since Phase 1 scaffold).
5. **Editing UI:** new `ResumeEditor.tsx` in `src/components/resume/` (or `jobs/`), reusing `EditableList`/`StringList`/`primitives.tsx` like every other form in the app. Any bullet whose `sourceMap` entry has zero `atomIds` is visibly flagged (per PRD §7/§13 — "no generated resume line lacks a sourceMap entry, or it is visibly flagged"). This alone isn't sufficient, though -- see "over-claiming" under Open decisions below; a bullet can cite real, non-empty `atomIds` and still overstate what they support.
6. **Export:** an HTML resume template + print CSS (`@media print`), rendered from `ResumeContent`, "Export PDF" triggers `window.print()`. No new dependency needed.

## Open decisions (need user's call before implementing)
- **Regeneration vs. versions:** does a posting have exactly one resume generation (regenerating overwrites, like matching's "Re-run" pattern), or a history of generations to pick from? PRD §4.8 says "keep a local history of postings and generations per job" — suggests multiple, kept.
- **Bullet source scope:** should generation only draw from atoms already used as requirement evidence + additionalInfo, or the full profile (with the model free to pick anything, same anti-fabrication check applied after)? Leaning toward confirmed-evidence-only pool since it's stricter and reuses matching's work, but it means unmatched-but-relevant profile content never surfaces.
- **Resume length/section control:** any target length, or does the model include everything with matched evidence?
- **Over-claiming beyond empty-atomIds:** the "flag if zero atomIds" check (item 5) catches unevidenced bullets but not overstated ones -- e.g. a bullet claiming "3+ years of Java experience" citing a real but bare `Skills: Java` atom (no duration in the atom text at all) would pass that check while still overstating the evidence, exactly the failure mode found in Phase 3's matching verification. Needs either a tighter generation-prompt constraint (don't state a duration/quantity/scope that isn't literally present in the cited atoms' text) or a lightweight second pass that checks each bullet's claim against its cited atoms' actual text, not just checking the id list is non-empty.

## Default assumptions if not otherwise directed
- Atom-id sourceMap (not free text).
- Evidence-only atom pool (requirement matches + additionalInfo).
- One generation per posting, overwritten on regenerate, matching the matching screen's "Re-run" pattern.
